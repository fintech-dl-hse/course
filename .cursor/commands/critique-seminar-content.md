# Критика контента семинара (режим: только план)

Проанализируй ноутбук семинара `$ARGUMENTS`. Учитывай фон студентов и план курса. **Режим: только план** — не вноси правки в ноутбук, только сформируй и выведи структурированный план улучшений.

## Порядок действий

1. Прочитай ноутбук `$ARGUMENTS`.
2. Прочитай `README.md` — определи номер семинара, какие лекции и домашки студенты уже прошли к этому моменту.
3. Используй раздел «Seminars Overview» из `CLAUDE.md` как якоря тем и стиль оформления (он уже загружен в контекст).
4. Сформируй отчёт по шаблону ниже.

## Контекст: фон студентов

- На курсе люди с разными бэкграундами; материал не должен предполагать знаний, которые ещё не даны в курсе.
- Не требуй от студентов знаний из тем, которые идут позже в таблице «Примерный план» из README.

## Что анализировать

1. **Соответствие плану курса**
   - Нет ли ссылок на концепты/лекции, которые ещё не читались?
   - Нет ли опережения (например, CNN до блока CV, attention до NLP)?

2. **Педагогика «снаружи внутрь» (Outside-In)**
   - Сначала ли идёт практика (как использовать), затем объяснение «как устроено», затем закрепление?
   - Нет ли перегрузки теорией до первого запускаемого примера?

3. **Сложность и доступность**
   - Понятен ли материал студенту с «нулевым» бэкграундом по этой теме в рамках курса?
   - Нет ли неочевидных скачков (вводятся ли термины и обозначения перед использованием)?
   - Соответствует ли объём кода и формул длине семинара?

4. **Recap и связь с прошлым**
   - Есть ли краткий recap предыдущего семинара/лекции?
   - Связь с домашками и лекциями явно указана?

5. **Блиц и вопросы**
   - Вопросы проверяют понимание именно этого семинара?
   - Нет ли вопросов по темам, которые на семинаре не разбирались?
   - Ответы на вопросы в Блице должны быть в спойлерах (см. «Seminars Overview» в CLAUDE.md).

6. **Домашки**
   - Есть ли блок «Домашки» с обзором: что сдавать, на что обратить внимание, типичные ошибки?
   - Соответствует ли описание домашек списку из README для этого момента курса?

7. **Ясность и стиль**
   - Один концепт на ячейку/подраздел?
   - Код с type hints и короткими docstrings для переиспользуемых функций?
   - Математика: `$...$` и `$$...$$`, без `\(...\)` / `\[...\]`?

8. **Дубли разделов и порядок тем/ячеек**
   - Нет ли повторяющихся по смыслу разделов (одинаковая тема под разными заголовками, один и тот же концепт в разных местах)?
   - Логичен ли порядок: заголовки и ячейки идут в последовательности, удобной для восприятия (определение → пример → упражнение)?
   - Нет ли «разрывов»: тема начата, потом неожиданно другая тема, затем возврат к первой без явной связи?
   - Соответствует ли порядок ячеек заявленной структуре семинара (план в начале ноутбука)?

## Формат вывода (только план, без правок)

### 1. Контекст семинара
- Номер/тема семинара, соответствующая лекция и домашки по плану.
- Что студенты уже прошли к этому занятию (кратко).

### 2. Сильные стороны
- 2–4 пункта: что в контенте уже хорошо согласовано с планом и педагогикой.

### 3. Замечания и риски
- По каждому пункту из «Что анализировать»: конкретные замечания с указанием ячеек/секций (по названию заголовка или фрагменту текста). Если замечаний нет — кратко написать «Без замечаний».

### 4. План улучшений
- Нумерованный список конкретных шагов: что добавить, переставить, упростить или уточнить. Для каждого шага — секция/ячейка и желаемое изменение (без правки кода/текста в ноутбуке).

### 5. Рекомендации по сложности и фону
- Что может быть сложно для студентов с разным бэкграундом.
- Предложения: дополнительные пояснения, ссылки на лекцию/README, сокращение или перенос материала.

**Важно:** не вызывай инструменты редактирования ноутбука — только анализ и текстовый вывод плана.
